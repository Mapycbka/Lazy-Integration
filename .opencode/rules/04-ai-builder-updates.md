# AI Builder: проверка и применение обновлений доставки пайплайна

Условное имя продукта для пользователя: **AI Builder** (поставка managed-файлов OpenCode Pipeline в проект).

## Цели

- Держать установленную версию (см. `docs/specs/opencode-pipeline.lock.json` → `pipelineVersion`) сопоставимой с доступной в **доверенном npm Registry** для пакета `@beeline/yellowbe-opencode-pipeline`.
- **Не** выполнять проверку на каждом переходе между этапами графа (`request-analyst` → `designer` → …). После возврата `Task` от stage agent и выборе следующего stage **не** запускать проверку обновления только по этому факту.
- Не блокировать основной сценарий задачи: при медленной сети, таймауте или ошибке registry — мягкий отказ (продолжить на текущей версии); синхронная попытка проверки укладывается в бюджет времени CLI (`check-update --budget-ms`, по умолчанию 450 ms).
- Не перезаписывать пользовательские артефакты вне списка managed / `userOwnedPaths`; применение обновления — только через тот же механизм, что и managed-update (`safe-update` или последовательность `diff` → `update` из опубликованного пакета).

## Источник истины

- Состояние: `docs/specs/pipeline-state.yaml` → блок **`ai_builder`**.
- CLI (после установки пакета или через `npx`):  
  `npx --yes @beeline/yellowbe-opencode-pipeline@latest check-update --target <корень проекта> --json [--budget-ms 450]`  
  `npx --yes @beeline/yellowbe-opencode-pipeline@latest safe-update --target <корень проекта> [--mcp-config …] [--json]`

## Уровни политики UX (`ai_builder.update_policy_level`)

### Уровень 1 — Тихая проверка

- Выполни проверку при срабатывании триггера (ниже).
- Если версии совпадают или обновления нет (`status`: `up_to_date`) — **ничего не сообщай** пользователю.
- Запиши результат в `ai_builder.last_update_check_at`, `last_update_check_status`, при успехе — `published_latest_seen`.
- При `failed` или `skipped_timeout` на уровне 1 — **без сообщения** (только запись в состоянии).

### Уровень 2 — Мягкое уведомление

- Как уровень 1, плюс при `update_available` (и если не действует `notify_snooze_until`) — **одно короткое сообщение** простым языком, без команд, путей к файлам и терминов вроде registry/npm.

**Пример тона:**

«Доступно обновление AI Builder с исправлениями. Применить сейчас или после текущего шага?»

**Варианты ответа пользователя (зафиксируй в `last_stage_result` или продолжение диалога):**

- **Сейчас** — выполни безопасное применение (см. «Применение»).
- **Посже** — не применяй; продолжи задачу; следующая проверка по интервалу/триггерам.
- **Не предлагать сегодня** — выставь `ai_builder.notify_snooze_until` на конец локальных суток пользователя (или +24 ч от текущего момента, если дата неизвестна), затем продолжи задачу.

### Уровень 3 — Авто-применение в безопасных условиях

- После тихой проверки, если `update_available` и **нет** необходимости спрашивать пользователя по другим основаниям:
  - Запусти **`safe-update`** (или эквивалент: опубликованный пакет `diff` без конфликтов → `update`).
  - При успехе обнови запись в `ai_builder` и продолжи основную задачу **без длинного отчёта** (одна нейтральная фраза допустима по политике организации).
- Если `safe-update` / `diff` сообщает о **конфликте** managed-файлов или нестандартном состоянии lock — **не применяй молча**: перейди к формулировкам уровня 2 или одному экрану «нужно решение» простым языком + опция обратиться к разработчику/поддержке.

## Когда выполнять проверку (триггеры)

Выполняй логику AI Builder **только в начале обработки нового сообщения пользователя** addressed to `pipeline-orchestrator`, если выполняется хотя бы одно из условий ниже. **Не** запускай её заново только потому, что завершился внутренний `Task` stage agent и ты выбираешь следующий stage в том же пользовательском ходе.

### Автоматические

1. **Старт работы с проектом в сессии (эвристика):** первое пользовательское сообщение после того, как `orchestrator_status` был `idle` или пустой `last_mode`, **или** смена календарного дня относительно `last_user_activity_at`. (При появлении в продукте нативного session id можно уточнить триггер без смены смысла правила.)
2. **Интервальная дедупликация:** с момента последней успешной проверки (`last_update_check_at` при статусе не `failed` / не только timeout — трактуй успех как получен любой однозначный ответ registry: `up_to_date`, `update_available`, `no_lock`) прошло **≥ `ai_builder.check_interval_hours`** часов — при следующем **пользовательском** сообщении выполни уровень 1 (не чаще одного раза за интервал).
3. **Возобновление после простоя:** с `ai_builder.last_user_activity_at` прошло **≥ `ai_builder.idle_resume_hours`** часов — при следующем пользовательском сообщении одна проверка уровня 1 с учётом интервала из п. 2 (не дублировать две проверки подряд в одном сообщении).

Обновляй **`ai_builder.last_user_activity_at`** на каждое новое пользовательское сообщение к оркестратору (ISO-8601).

### Явный запрос пользователя (must have)

Распознавание намерений (расширяй список по мере необходимости):

| Намерение | Примеры формулировок |
|-----------|---------------------|
| Только проверка | «Проверь обновление пайплайна», «Проверь обновления AI Builder», «Есть ли новая версия AI Builder?» |
| Проверка и применение | «Обнови AI Builder», «Поставь последнюю версию пайплайна», «Обнови доставку пайплайна» |

Поведение:

- **«Проверь»** — выполни `check-update` с приоритетом над кэшем интервала; не чаще **одного раза в минуту** на проект в одной сессии: если `session_forced_check_at` был менее 60 секунд назад — пропусти повторный сетевой вызов и используй последний известный результат (или коротко сообщи, что проверка недавно уже выполнялась).
- **«Обнови»** — выполни проверку (с тем же анти-спамом 1 минута); затем если есть новая версия — выполни **`safe-update`**; при конфликте — уровень 2 и одна фраза объяснения без технических деталей.

## Применение обновления

Используй только опубликованный пакет:

```bash
npx --yes @beeline/yellowbe-opencode-pipeline@latest safe-update --target . --json
```

При необходимости передай тот же `--mcp-config`, что принят в проекте. Не скачивай и не выполняй код произвольных URL.

После успешного применения перечитай при необходимости локальные промпты/rules; **не** изменяй содержимое путей из `userOwnedPaths` средствами обновления.

## Ошибки и NFR

- Если `check-update` вернул `skipped_timeout` или `failed` — установи `last_update_check_status`; при уровне **2 или 3** допустима **одна** короткая фраза: «Проверка обновлений недоступна, работаем на текущей версии.» При уровне **1** — без сообщения.
- Не повторяй синхронную проверку в том же ходе после таймаута.

## Копирайт (русский UI)

**Обновление доступно (уровень 2):**

«Доступно обновление AI Builder с исправлениями. Применить сейчас, после текущего шага или не предлагать сегодня?»

**Конфликт при применении:**

«Обновление AI Builder сейчас не удалось применить автоматически: есть локальные изменения в файлах доставки. Нужно решение вручную или помощь разработчика. Продолжаем задачу на текущей версии?»

**Fallback сеть:**

«Проверка обновлений недоступна, работаем на текущей версии.»
