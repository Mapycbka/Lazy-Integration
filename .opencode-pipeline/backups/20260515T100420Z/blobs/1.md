# Формирование конфигурации БС

Настольное приложение на Python + PySide6 для формирования конфигурационного Excel-файла базовой станции на основе:

- файла ДИ
- файла ТС
- шаблона из папки `templates/<регион>/<тип_BBU>/`

Приложение работает с копией шаблона, не изменяет оригинал, пишет лог в GUI и в `logs/app.log`, поддерживает `xlsx` и `xlsm`, а также подготовлено к сборке в Windows `.exe`.

## Что умеет приложение

- выбор файла ДИ
- выбор файла ТС
- галочка полного пропуска загрузки ТС-параметрики
- выбор типа площадки, региона, типа BBU и шаблона из выпадающего списка
- выбор софта BTS и BSC
- выбор порта транспортной платы
- ввод одного или нескольких слотов/плат
- ввод портов в форматах `0-2`, `0 1 2`, `0,1,2`, `0, 1, 2`
- всплывающий выбор `плата - порт` для LTE-цепочек
- ручной ввод ТС-параметров, если ТС-файл неполный или нестандартный
- пропуск ТС с сохранением DI-derived полей и очисткой транспортных полей
- клонирование строк шаблона с сохранением стилей и формул
- логирование статусов, предупреждений и ошибок

## Ключевая логика

### Шаблон

- Шаблон не загружается вручную.
- Пользователь выбирает:
  - тип площадки
  - регион
  - тип BBU
  - шаблон
- Список шаблонов формируется автоматически из:

```text
templates/<регион>/<тип_BBU>/
```

- При обработке создается рабочая копия шаблона во временной папке `temp/`.

### NE Name и замены по книге

Из ДИ берется:

- лист `Конфигурация сайта`
- столбец `Ne Name(name@OSS)`

Далее:

- на листе `NE Version` заполняется только одна строка
- старые значения `NE Name` заменяются по всей книге
- выбранные версии софта BTS и BSC заменяются по всей книге

Для BTS сохраняется исходный префикс модели:

- `BTS3900_5900 ...` остается `BTS3900_5900`
- `BTS5900 ...` остается `BTS5900`

Меняется только версия `V100R...`.

### ТС-параметры

ТС читается не только с листа `ip-план`, но и с допустимых вариантов имени, например:

- `ip-план`
- `ip план`
- `IP plan`
- `IP-Plan`
- `IP BTS`
- `IP BS`
- `ИД БС-ТС`
- `Заявка РЦМ`

Если в ТС нет нужного листа, шапки или столбцов, приложение предлагает:

- ввести ТС-параметры вручную
- пропустить ТС
- отменить формирование

Этот же сценарий используется, если лист ТС найден, но не содержит подходящих заголовков для чтения транспортных параметров.

При пропуске ТС:

- поля, которые вычисляются из ДИ, заполняются
- чисто транспортные поля очищаются

Также ТС можно пропустить заранее через галочку в главном окне:

- файл ТС можно не выбирать
- формирование идет только по ДИ
- поля, которые должны прийти из ТС, остаются пустыми

Поддерживаются алиасы колонок ТС. Для полей:

- `OMCH`
- `OMCH NH`
- `ABIS`
- `ABIS NH`
- `S1`
- `S1 NH`

в шаблон записывается только IPv4-адрес, даже если в ТС в ячейке есть маска или дополнительный текст.

Для полей:

- `OMCH Vlan`
- `ABIS Vlan`
- `S1 Vlan`

в шаблон записывается только сам VLAN, например:

- `3702 (vprn 71007000)` -> `3702`

### Base Station Transport Data

Заполняются:

- `*Name`
- `*BTS Name`
- `*eNodeB Name`
- `OMCH`
- `OMCH NH`
- `OMCH Vlan`
- `ABIS`
- `ABIS NH`
- `ABIS Vlan`
- `S1`
- `S1 NH`
- `S1 Vlan`
- `*eNodeB ID`
- `ADJNODE (ID)` или `ID`
- `SCTPLNK(BSC) (ID1)` или `ID1`
- `Port`

Заголовки поддерживаются и как обычные столбцы, и как подстолбцы второй строки шапки.

### 2G / 4G / RRU / Sector

Из ДИ строятся:

- `GSM Cell`
- `GTRXGROUP`
- `LTE Cell`
- `RRUCHAIN(NODE)`
- `RRU(NODE)`
- `SECTOR(NODE)`
- `SECTOREQM(NODE)`

Правила:

- `RRUCHAIN(NODE)` и `RRU(NODE)` зависят от уникальных значений `RRU` в ДИ
- `SECTOR(NODE)` зависит от количества секторов в `2G_BTS_Data` и `4G_Data`
- если один и тот же `RRU` встречается и в 2G, и в 4G, это считается общим физическим RRU
- если у общего `RRU` LTE часть `2T2R`, то в `RRU(NODE)` такой RRU формируется как `4T4R`

Для общих 2T2R-секторов поддержано распределение антенн по шаблону, например:

- `90 -> R0A,R0B`
- `91 -> R0A,R0C`
- `92 -> R0B,R0D`

Эта логика применяется и в:

- `SECTOR(NODE)` -> `Sector Antenna`
- `SECTOREQM(NODE)` -> `Sector Equipment Antenna`

При этом для отдельных кейсов, где 2G висит на общем 4T4R-RRU, но не делится между несколькими 2G-секторами, 2G-оборудование может использовать поднабор `R0A,R0C`.

### Sector Equipment ID

Поддержано формирование `Sector Equipment ID`:

- для 2G префикс `2`
- для 4G префикс `4`
- далее трехзначный `chain/sector`
- последняя цифра по правилу

Для `1800/2100` поддержана отдельная логика:

- `1800 -> ...0`
- `2100 -> ...1`

Пример:

- `42000`, `42010`, `42020`
- `42001`, `42011`, `42021`

## Поддерживаемые листы и устойчивость к названиям

Приложение не завязано на одно точное имя листа. Поддерживаются алиасы для:

- листов ДИ
- листов ТС
- листов шаблона

Также поддерживаются:

- сдвинутая шапка
- подстолбцы во второй строке
- алиасы заголовков
- варианты `ID` / `ID1` вместо полных имен
- варианты `*Cell ID` / `*LocalCellID`

## Запуск

### macOS / Linux

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python3 main.py
```

Быстрый запуск для macOS:

```bash
./run_macos.command
```

### Windows

Быстрый запуск:

```bat
run_windows.bat
```

или:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\run_windows.ps1
```

Скрипт:

- создаст `.venv`, если нужно
- установит зависимости
- запустит приложение

## Сборка Windows EXE

Сборка выполняется на Windows:

```bat
build_windows_exe.bat
```

или:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\build_windows_exe.ps1
```

Результат:

```text
dist/BS_Config_Generator.exe
```

Для запуска готового `.exe` Python на пользовательском ПК не нужен.

## GitHub Actions

В проекте подготовлен workflow:

```text
.github/workflows/build-windows-exe.yml
```

Он:

- находит папку проекта в репозитории
- поднимает Windows runner
- ставит зависимости
- собирает `.exe`
- загружает его как artifact

## Ограничения и расширение

- В проекте уже заложены алиасы листов и колонок, поэтому новые варианты имен обычно добавляются точечно в конфиг, без переписывания логики.
- Некоторые реальные шаблоны могут содержать нестандартные объединенные ячейки, формулы или особые служебные листы. Для таких случаев лучше проверять результат на конкретном шаблоне.
- Если у заказчика появится новый тип распределения антенн или новый формат ТС, его удобнее добавлять в `core/constants.py`, `core/transforms.py` и `core/rru_builder.py`.
